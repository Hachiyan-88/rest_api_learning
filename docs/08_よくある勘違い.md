# よくある勘違い集（もう一段理解を深める）

このページでは、
>REST APIとJSONに関して、初心者がやりがちな勘違い

を敢えて整理してみる。

勘違いを潰すことで、理解をもう一段深めることができる。

「なんとなく分かった気」で進まないために、ここで整理しよう思う。

---

## ❌勘違い①：REST＝JSON

これはかなり多いらしい。

でも実際は：

- REST → 設計思想
- JSON → データ形式

RESTはデータ形式を決めていない。
>REST APIでJSONがよく使われているだけ。

ここを分離できるとかなりOK

---

## ❌勘違い②：JSONはオブジェクトそのもの

JSONは
>ただの文字列（テキスト）

プログラム内のオブジェクトとは別物扱い！

- JSON → 文字データ
- オブジェクト → メモリ上の実体

変換（シリアライズ / デシリアライズ）が必要

---

## ❌勘違い③：POSTは「追加専用」メソッド

結構よくある思い込みのパターン。

RESTの思想では：

- POST → 新規作成にもよく使う

でも厳密には「処理を依頼する」意味も持つ。

PUTやPATCHとの違いを理解することが重要になる。

---

## ❌勘違い④：URLに動詞を書けばRESTっぽい

例：

```JSON
POST /createUser
GET  /getUsers
```

一見わかりやすいが、RESTの思想とはズレている。

RESTでは：

```JSON
POST /users
GET  /users
```

リソース（名詞）で設計しよう。

---

## ❌勘違い⑤：Controllerに全部書けばいい

小さいアプリだと動いてしまう。

でも設計としては：

- Controller → 受付
- Service    → ビジネスロック
- Repository → データ操作

役割分担が重要になる。

後から拡張できなくなる原因になる可能性すらある。

---

## ❌勘違い⑥：エラーは「なんか壊れている」って認識

400や500のエラーが出ると焦るよね。

実際は多くの場合：

- JSONの構造ミス
- 型の不一致
- null問題

どこで止まっているか考えるようにすることが重要。

---

## ❌勘違い⑦：動いている＝正しい設計

動くコードと、そのコードが良い設計かどうかは別。

- RESTの思想に沿っているのか。
- 責務が分離されているか。
- 将来変更しやすいか。

この辺りまで考えられるようになると、色々楽になるらしい。

---

### 自分用メモ

- 混同している概念は分けて考える
- 「なぜ？」を考える
- 動くだけで満足しないようにする

---

## まとめ

勘違いはわるいことではない。

でも、
>放置すると理解が止まってしまう。

これだけが良くないらしい。自分も同意です。

このページで挙げたポイントを意識できれば、
REST APIの理解はかなり安定するような気がして整理してみました。
