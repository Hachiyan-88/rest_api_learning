# JSONの基礎（データの形を理解する）

このページでは、
**REST APIでほぼ必ず使われるJSON形式**について、
自分があとから読み返しても理解できるように、丁寧に整理していくつもりです。

ここは「なんとなく」で流しがちな部分らしい。
でも、ここをちゃんと理解するとAPIの見え方が一段クリアになる。

---

## まず一言でいうと

JSONとは：
> データを書くためのルール（フォーマット）

プログラミング言語ではない。
あくまで「データの見た目の決まり」。

---

## なぜJSONが使われるのか？

理由は結構シンプル。

- 人間が読んでもある程度わかる
- 記述がシンプルで軽い
- ほとんどの言語で扱える
- Web（特にJavaScript）と相性がいい

昔はXMLが多く使われていたが、
よりシンプルで扱いやすいJSONが主流になっていった。

---

## JSONの基本構造

JSONには主に2つの要素がある。

- オブジェクト
- 配列

この2つを理解できれば、JSONはほぼ読めるようになる。

---

## ① オブジェクト

```json
{
  "name": "Taro",
  "age": 25,
  "isStudent": true
}
```

### ①特徴

- `{}` で囲む
- 「キー : 値」のセットで構成される
- キーは必ずダブルクォートで囲む

### ①イメージ

> 「ラベル付きの箱」

- name というラベルの箱に "Taro"
- age というラベルの箱に 25

キーがラベル、値が中身。
この対応関係が超重要。

---

## ② 配列

```json
{
  "users": [
    { "name": "Taro" },
    { "name": "Hanako" }
  ]
}
```

### ②特徴

- `[]` で囲む
- 同じ種類のデータが並ぶことが多い

### ②イメージ

> 「同じ形の箱が並んでいる棚」

オブジェクトが何個も並ぶケースは実務で頻出。
APIレスポンスでよく見る形。

---

## JSONで使えるデータ型

- 文字列（"text"）
- 数値（123）
- 真偽値（true / false）
- null
- オブジェクト
- 配列

※ 日付型は存在しない（文字列で表現するのが一般的）

例：

```json
{
  "createdAt": "2026-02-13T10:00:00"
}
```

---

## ネスト構造（入れ子）

JSONは入れ子にできる。

```json
{
  "user": {
    "name": "Taro",
    "address": {
      "city": "Tokyo"
    }
  }
}
```

最初は見づらいが、

- どの `{}` がどこに対応しているか
- インデント（字下げ）を意識する

これだけでかなり読みやすくなる。

---

## JSONとプログラムのオブジェクトの違い

ここは結構混乱ポイント。

### JSON

- ただのテキスト
- データの表現形式

### JavaやJavaScriptのオブジェクト

- メモリ上の実体
- メソッド（処理）を持てる

👉 JSONは「文字列」。
👉 オブジェクトは「実体」。

JSONはそのままでは動かない。
プログラムに変換して初めて使える。

---

## REST APIとJSONの関係

RESTはデータ形式を決めていない。

でも実際には：

> ほぼJSONが使われている

だから「REST = JSON」と思いがち。
でも正確には違う（詳しくは06章）。

```ini
REST = 通信ルール
JSON = データ形式
```

```csharp
[REST通信]
      ↓
{ "name":"Taro" }
```

---

## リクエストとレスポンスの例

### リクエスト

```http
POST /users
Content-Type: application/json
```

```json
{
  "name": "Taro",
  "age": 25
}
```

### レスポンス

```json
{
  "id": 1,
  "name": "Taro",
  "age": 25
}
```

👉 JSONを送ってJSONが返る。
これがREST APIでよく見るやり取り。

---

## よく起こるエラー例

- カンマの付け忘れ
- ダブルクォートの付け忘れ
- 最後の要素に余計なカンマ

例：❌

```json
{
  "name": "Taro",
}
```

JSONはルールに厳しい。
ちょっとした記述ミスでエラーになる。

---

## Content-Typeとは？

HTTPヘッダーにある情報。

```http
Content-Type: application/json
```

これは：

> 「この通信の中身はJSONですよ」

と宣言している。

サーバー側はこれを見て、JSONとして処理する。

---

## 自分用メモ

- JSONはただのテキスト
- オブジェクトとの違いは超重要
- ネスト構造は慣れが必要
- エラーは大体カンマかダブルクォート

---

## まとめ

- JSONはデータ形式（フォーマット）
- オブジェクトと配列が基本
- REST APIではほぼ標準
- 読めるようになるとAPI理解が一気に進む
