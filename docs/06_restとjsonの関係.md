# RESTとJSONの関係（図で整理してみる）

このページでは、
>RESTとJSONはどういう関係なの？

を、図でイメージできるように整理する。

「REST＝JSON」と思いがちだけど、実は少しズレた認識らしい。

ここを分解していくと、API全体の構造がかなりクリアになるみたい。

---

## まず結論

- REST = 設計思想（ルール）
- HTTP = 通信の仕組み
- JSON = データの書き方

この3つは、**役割がはっきり分かれている**。

---

## 全体像（まずは俯瞰で）

```JSON
[クライアント]
    ↓①HTTPで通信
[サーバー]
    ↑

中身（データ形式）＝JSON
設計ルール　　　　＝REST
```

もう少し分解してみると・・・

```JSON
REST → APIの設計ルール
HTTP → データを運ぶトラック
JSON → トラックに積まれる荷物
```

全部セットで使われることが多めだけど、それぞれの役割は別物

---

## RESTは「設計思想」

RESTは：
>リソースをどう扱うか？

を定めた考え方。

例えば：

```JSON
GET     /users      →一覧取得
GET     /users/1    →1件取得
POST    /users      →新規作成
PUT     /users/1    →更新
DELETE  /users/1    →削除
```

ここで決まっているのは：

- URLの設計
- HTTPメソッドの使い方

**データ形式までは決めていない。**

---

## JSONは「データ形式」

JSONは：
>データをどう書くか？

を決めるフォーマット。

例：

```JSON
{
    "name":"Taro",
    "age":25
}
```

これはあくまでも
>データの見た目のルール

RESTとは役割が違う。

---

## なぜREST × JSONが定番になったのか？

理論上、RESTは

- XMLでもOK
- CSVでもOK

でも実際はJSONがほぼ標準になっている。

理由：

- 軽い
- 書きやすい
- JavaScriptと相性が良い
- WEB開発と相性が良い

結果として：
>REST API + JSON

が事実上の定番になった。

---

## 実際の通信を図で見てみよう

### 例：ユーザー登録

```JSON
①クライアント
    ↓ POST /users（HTTP）

②リクエストボディ（JSON）
    {
        "name":"Taro",
        "age":25
    }

③サーバー処理（REST設計に沿う）

④レスポンス（JSON）
    {
        "id":1,
        "name":"Taro",
        "age":25
    }
```

ポイントは：

- 通信方法＝HTTP
- 設計思想＝REST
- 中身    ＝JSON

役割が完全に分かれていること。

---

### Content-Typeの意味（確認用）

HTTPヘッダーにある：
>Content-Type: application/json

これは：
>「この通信の中身はJSONですよ」

という宣言。

RESTだからJSONなのではない！

「今回はJSONを使っています」というだけ。

---

## 3層で整理すると理解しやすいかも

```JSON
①設計層     →   REST
②通信層     →   HTTP
③データ層   →   JSON
```

この構造で考えられるようになると、API全体がかなりクリアに整理できる。

---

## 自分用メモ

- RESTとJSONは別物だということ
- HTTPはただの通信手段らしい
- 3層構造で考えると混乱しにくい
-「REST = JSON」と思わないこと

---

## まとめ

- RESTは設計思想の役割
- HTTPは通信プロトコル
- JSONはデータ形式

この3つを分けて考えられるようになった方が良さそう。
